# Product Diagnosis Dictionary

## 1. 文档目标

本文档定义 `commerce-video-diagnosis` 体系中的**商品诊断输入口径**。

它只回答一件事：调用方在进入商品诊断前，必须按什么业务结构提供 `product_info`，以及每一层字段分别承载什么语义。

本文档作为以下对象的统一业务口径：
- `SKILL.md` 中的 payload 说明；
- `docs/input_output_schema.md` 中的商品字段解释；
- machine-readable JSON Schema 的业务依据；
- 产品、研发、QA、运营对 `product_info` 的统一定义。

本文档**不承担**：
- 数据库 SSOT 查询逻辑；
- LLM Prompt 细节；
- 视频理解输出字段定义；
- HEC 标签本身的业务解释。

---

## 2. 总原则

### 2.1 `product_info` 必须由调用方显式提供

商品诊断禁止依赖运行时自行猜商品、补商品或从视频里反推完整商品主档。

调用方必须显式传入 `product_info`。原因如下：
- 避免 `source_product_id` 与真实投放商品不一致；
- 避免内部 SSOT 缺失或误命中导致诊断漂移；
- 保证视频理解、商品诊断、脚本生成三个阶段使用同一份商品事实；
- 便于协议层做强断言与 Crash Early。

### 2.2 `product_info` 采用固定三层结构

`product_info` 必须拆成以下三层：
1. **4.1 基础商品信息**：最小识别层，回答“这是什么商品”；
2. **4.2 完整商品信息**：业务表达层，回答“主要卖给谁、核心差异点是什么”；
3. **4.3 商品事实向量**：诊断推理层，回答“该商品在任务、门槛、风险上的结构化事实是什么”。

### 2.3 三层职责不可混淆

- 4.1 解决“商品身份识别”；
- 4.2 解决“商品业务表达”；
- 4.3 解决“商品诊断推理输入”。

禁止出现以下混用：
- 把人群、差异化卖点塞进基础层；
- 把推理结论、策略结论塞进完整信息层；
- 把 HEC 标签、脚本策略、视频判断结果塞进事实向量层。

---

## 3. 顶层结构

推荐 `product_info` 结构如下：

```json
{
  "product_info": {
    "basic_product_info": {
      "leaf_category": "定型喷雾",
      "shop_name": "示例旗舰店",
      "item_name": "示例商品",
      "price": 79.9,
      "core_selling_points": ["快速蓬松定型", "无胶感不粘腻"]
    },
    "full_product_info": {
      "target_people": ["头发容易贴头皮的人群", "需要快速打理发型的男性"],
      "differentiator": "一喷一吹就能快速做出蓬松发型，且无胶感不粘腻"
    },
    "product_fact_vector": {
      "product_task": "降本增效/懒人替代",
      "cognitive_attribute": "低认知门槛",
      "frequency_attribute": "高频",
      "trust_attribute": "中",
      "price_attribute": "中",
      "endorsement_attribute": "无明确权威背书"
    }
  }
}
```

说明：
- 字段名可在工程实现中微调，但三层语义必须保持稳定；
- 若某层缺失，协议应直接报错，不做静默兼容；
- 若字段值证据不足，应由调用方补齐，不能让模型自行脑补。

## 4. 三层字段定义

## 4.1 基础商品信息（Basic Product Info）

### 4.1.1 定义

基础商品信息是商品诊断的最小识别层，用于稳定回答：
- 这是什么类目的商品；
- 谁在卖；
- 商品叫什么；
- 大致价格是多少；
- 对外主卖点如何表述。

### 4.1.2 字段清单

| 字段 | 类型 | 必填 | 含义 |
|---|---|---:|---|
| `leaf_category` | string | 是 | 叶子类目，必须是足够具体的商品类目 |
| `shop_name` | string | 是 | 店铺名称，必须透传调用方原始值 |
| `item_name` | string | 是 | 商品标题或商品名 |
| `price` | number / string | 是 | 当前对外使用的价格口径 |
| `core_selling_points` | array of string | 是 | 商品对外主卖点列表 |

### 4.1.3 字段定义与边界

#### `leaf_category`

定义：商品所属的**最细粒度叶子类目**。

作用：
- 决定诊断所处品类语境；
- 约束后续任务类型、阻力类型、对比对象判断；
- 防止不同大类商品套用错误诊断模板。

正例：
- `定型喷雾`
- `洗发水`
- `防晒喷雾`

反例：
- `个护`
- `美妆`
- `日用品`

#### `shop_name`

定义：销售主体的店铺名称。

作用：
- 用于后续品牌/白牌、信任存量、店铺侧路由判断；
- 作为实体校验输入。

边界：
- 必须透传原始文本；
- 禁止模型自动补全、繁简转换或主观归一化；
- 若调用方拿不到店铺名，应直接报缺失，不得猜测。

#### `item_name`

定义：商品标题或最稳定的商品名称。

作用：
- 提供商品身份识别；
- 为卖点抽取、任务归因提供文本锚点。

边界：
- 可以是电商标题，也可以是规范化商品名；
- 但必须真实对应当前视频所售商品；
- 禁止用泛词代替，如“示例好物”“某爆款”。

#### `price`

定义：当前诊断所使用的价格口径。

作用：
- 为价格属性、价格阻力、性价比判断提供事实输入。

边界：
- 推荐使用 number；若外部系统只能提供 string，也必须保证可解析；
- 必须是单商品对外成交语境下的价格，不要混入划线价、满减后估算价、套装总价；
- 若价格未知，不得填默认值。

#### `core_selling_points`

定义：调用方认可的商品核心卖点列表。

作用：
- 作为商品对外表达层的原始事实；
- 为 `differentiator` 与后续脚本表达提供事实锚点。

边界：
- 必须写商品卖点，不写创意策略；
- 一般 2～5 条为宜；
- 必须尽量可落到商品本身，而不是空泛形容。

正例：
- `快速蓬松定型`
- `无胶感不粘腻`
- `控油清爽`

反例：
- `适合短视频爆单`
- `适合做对比演示`
- `建议走痛点开场`

---

## 4.2 完整商品信息（Full Product Info）

### 4.2.1 定义

完整商品信息是业务表达层，用于把“商品是什么”进一步补成“商品主要卖给谁、最值得记住的差异点是什么”。

这一层仍然属于**商品事实表达**，不是推理结论层。

### 4.2.2 字段清单

| 字段 | 类型 | 必填 | 含义 |
|---|---|---:|---|
| `target_people` | string / array of string | 是 | 商品主要目标人群 |
| `differentiator` | string | 是 | 商品最核心、最值得被记住的差异化卖点表达 |

### 4.2.3 字段定义与边界

#### `target_people`

定义：商品优先面向的购买/使用决策主体。

作用：
- 约束内容表达对象；
- 约束痛点与场景的语言口径；
- 为后续 JTBD、阻力、脚本表达提供对象锚点。

边界：
- 应优先描述“谁最需要它”，不是泛泛的人口学标签；
- 可用字符串，也可用数组；
- 若用数组，建议按主次列出 1～3 类核心人群。

正例：
- `头发容易贴头皮的人群`
- `需要快速打理发型的男性`
- `早上出门时间紧但想保持蓬松造型的人`

反例：
- `所有人`
- `年轻人`
- `爱美人士`

#### `differentiator`

定义：对商品核心差异化卖点的自然语言主表达。

作用：
- 作为商品最关键的对外记忆点；
- 为后续内容创作提供核心提炼句。

边界：
- 必须是商品卖点表达，不是策略表达；
- 必须尽量可被 `core_selling_points` 支撑；
- 禁止直接写 HEC 标签、脚本建议、平台策略。

正例：
- `一喷一吹就能快速做出蓬松发型，且无胶感不粘腻`
- `兼顾控油和清爽体感，适合日常快速整理发型`

反例：
- `适合走 H5 反常识与悬念`
- `建议用 E5 保姆级教程承接`
- `这是爆款表达`

---

## 4.3 商品事实向量（Product Fact Vector）

### 4.3.1 定义

商品事实向量是诊断推理层的结构化输入，用于把商品抽象成一组稳定可计算的业务事实。

这一层不是最终策略输出，而是后续商品诊断、内容路由、风险判断的**前置事实向量**。

### 4.3.2 字段清单

| 字段 | 类型 | 必填 | 含义 |
|---|---|---:|---|
| `product_task` | string | 是 | 商品要帮助用户完成的核心任务 |
| `cognitive_attribute` | string | 是 | 商品在认知理解维度上的属性 |
| `frequency_attribute` | string | 是 | 商品消费/使用频次属性 |
| `trust_attribute` | string | 内部推导 | 信任存量属性；由 `shop_name` 经品牌白名单（`brand_whitelist.csv` + `store_suffix_trust_dict.csv`）推导得出，调用方无需传入 |
| `price_attribute` | string | 内部推导 | 价格水位/价格阻力属性；由 `payload.engine_node.relative_price_level` 推导得出，调用方无需传入 |
| `endorsement_attribute` | string | 是 | 商品是否具有可用的权威/专业/官方背书 |

### 4.3.3 字段定义与边界

#### `product_task`

定义：商品要帮助用户完成的 JTBD 一级任务标签。

作用：
- 约束商品所在任务域；
- 为 JTBD、内容方向、对比逻辑提供一级锚点；
- 先判任务本体，再过前置门槛，最后才进入子标签分流。

> 这里的 JTBD 不是"给商品贴一个功能标签"，而是回答：用户为什么要在这个时刻买它、并期待它替自己完成什么任务。判断顺序固定为：先判任务本体，再过前置门槛，最后才进入子标签分流。

示例：

| 维度 | 一级任务 | 定义与判定锚点 |
|---|---|---|
| 功能任务 | 生存/运转维系 | 用于维持个体、家庭、设备或空间的基本正常运转。核心不是"更好"，而是"不出问题、不断档、能继续运转"。常见于刚需补给、基础维持、日常续命类商品。 |
| 功能任务 | 缺陷修复/冲突消除 | 用于修复已发生的破损、瑕疵、尴尬或摩擦，目标是把一个"已经出问题"的状态拉回可接受区间。凡是围绕修补、遮盖、止损、纠偏展开，优先落到这里。 |
| 功能任务 | 降本增效/懒人替代 | 用于把原本更麻烦、更费时、更费力或更费钱的动作，替换成更省事的方案。核心是"同一件事，用更低成本完成"，而不是单纯多一个新功能。 |
| 功能任务 | 物理安全与风险规避 | 用于预防客观伤害、规避可预见风险或降低事故概率。只要核心承诺是"别受伤、别出事故、别踩物理风险"，就应优先路由到这里，而不是混入普通修复或效率任务。 |
| 情绪任务 | 情绪安心/主观降险 | 商品提供的是主观层面的安定感、确定感、陪伴感或心理兜底。重点不在客观参数，而在"让我心里更踏实"。 |
| 情绪任务 | 新奇探索/瞬时刺激 | 购买动机来自尝鲜、猎奇、即时快感或感官刺激。核心是"想体验一下"，而不是长期维持或系统改善。 |
| 情绪任务 | 自我犒赏与秩序掌控 | 商品被当作一种奖励、自我安抚或生活秩序重建工具。既可能是"我值得拥有"，也可能是"我需要借它把生活重新理顺"。 |
| 社会任务 | 照护与责任履行 | 本质是在完成家庭、亲子、伴侣、宠物等关系中的照料与责任，不是表达身份，而是"我把该做的事做到位"。 |
| 社会任务 | 礼赠与关系表达 | 本质是在借商品完成送礼、致意、回礼、维护关系等动作。商品价值不只在物本身，更在它能否把关系表达准确送达。 |
| 社会任务 | 圈层认同（圈层归属/身份锚定） | 本质是在向某个已有共识的群体靠拢、证明"我是这个圈子的人"。核心前提是先在明确圈层语境，否则不能硬判。 |

边界：
- 必须使用 JTBD 任务字典中的**一级任务标签名称**；
- 先回答“用户为什么要在这个时刻买它、并期待它替自己完成什么任务”；
- 不能直接写成 HEC 或脚本动作；
- 不能把卖点句子、人群描述或策略判断混入 `product_task`；
- 应先判任务本体，再过前置门槛，最后才进入子标签分流。

##### JTBD 路由逻辑补充（功能型风险规避防降级）

> 背景：功能型风险规避商品（如驱蚊液、防虫/防护类）在 `difference_type=自身卖点陈述` 时，会被错误降级为「生存/运转维系」（基础供给型）。其真实 JTBD 是「物理安全与风险规避」。以下三条路由约束用于防止该误判。

1. **物理安全前置裁决必须覆盖防虫/防护类风险词（P0）**
   - 「物理安全与风险规避」的前置识别词表（业务字典 `product_diagnoser.jtbd.physical_safety_tokens`）不仅包含烫伤、触电、跌落、晒伤等显性伤害词，**还必须覆盖防虫/防护类风险词**：`蚊虫叮咬`、`叮咬`、`蚊虫`、`驱蚊`、`防蚊`、`防虫`、`驱虫`、`虫媒`、`驱避`、`瘙痒`、`红肿`、`过敏`。
   - 只要命中上述任一词，即在前置层（Stage A）唯一锁定为「物理安全与风险规避」，后续阶段不再改写 `primary_task`。

2. **`自身卖点陈述` 不再单独作为「生存/运转维系」的充分条件（P1）**
   - 仅凭 `difference_type=自身卖点陈述` + 无效率/易用语义，**不得**直接回落「生存/运转维系」。
   - 必须叠加硬约束：文本中**不存在任何风险/防护/缺陷语义**（即不命中 `physical_safety_tokens`、`protection_semantic_tokens`、缺陷信号）时，才允许回落「生存/运转维系」。
   - 一旦出现风险/防护语义，必须交回上游任务本体裁决（优先落「物理安全与风险规避」），禁止用基础维系兜底。

3. **桥接层对防护/风险类目自动注入隐式对比对象（P2）**
   - 当商品类目命中防护/防虫/防晒/安全等风险类目（业务字典 `product_diagnoser.jtbd.protection_risk_category_tokens`），且证据中存在风险/防护锚点时：
     - 即便调用方未提供 `comparison_object`，桥接层也**不得**直接把 `difference_type=自身卖点陈述` 清空为纯自身卖点；
     - 必须自动注入隐式对比对象（语义为「未使用任何驱避/防护产品」的旧方案基线），即 `comparison_object=同类旧方案`、`comparison_object_evidence_type=jtbd_inferred`，并将 `difference_type` 改写为 `风险降低`（`difference_domain=functional`）。
   - 若类目未命中防护风险词，或证据中无风险/防护锚点，则保持原协议：`自身卖点陈述` 时清空 `comparison_object` 并置 `comparison_object_evidence_type=null`。

#### `cognitive_attribute`

定义：商品在“用户是否容易理解其价值、是否容易理解怎么用”维度上的属性。

作用：
- 为认知门槛、教育成本、表达难度判断提供结构化输入。

示例值：
- `低认知门槛`
- `中认知门槛`
- `高认知门槛`

边界：
- 这是认知理解维度，不等于操作难度；
- 不要把“使用步骤多”直接写成高认知门槛，除非用户本身难以理解其价值或机制。

#### `frequency_attribute`

定义：商品在消费或使用频次维度上的属性。

作用：
- 为复购、购买惯性、替换成本、内容说服方式提供背景输入。

示例值：
- `高频`
- `中频`
- `低频`

边界：
- 频次只描述频次本身；
- 禁止从“高频”直接推导“试错成本高”或“操作门槛高”；
- 频次结论应基于品类常识或调用方事实，不得随意脑补。

#### `trust_attribute`

定义：商品在用户先验信任基础上的内部推导属性。

作用：
- 为信任阻力、背书需求、品牌侧表达强度判断提供输入。

推导来源：
- 由 `shop_name` 经品牌白名单（`brand_whitelist.csv` + `store_suffix_trust_dict.csv`）推导得出；
- 调用方无需传入。

示例值：
- `低`
- `中`
- `高`

边界：
- 该字段是系统内部诊断属性，不是调用方输入字段；
- 禁止调用方主观传入或覆盖；
- 推导结果必须以品牌白名单和店铺后缀字典裁决为准。

#### `price_attribute`

定义：商品在价格水位或价格阻力上的内部推导属性。

作用：
- 为价格敏感度、性价比表达、促销承接强度提供输入。

推导来源：
- 由 `payload.engine_node.relative_price_level` 推导得出；
- 调用方无需传入。

示例值：
- `低`
- `中`
- `高`

边界：
- 这是相对品类语境的价格属性，不是绝对价格高低；
- 不得脱离类目语境孤立判断；
- 不能把“客单价高”机械等于“高价格阻力”；
- 禁止调用方主观传入或覆盖。

#### `endorsement_attribute`

定义：商品是否具有可用的权威、专业、机构、达人或官方背书属性。

作用：
- 为可信度承接、专业表达、证据型内容策略提供输入。

示例值：
- `无明确背书`
- `有专业背书`
- `有品牌官方背书`
- `有达人背书`

边界：
- 必须是可被调用方明确说明的背书，不得凭空想象；
- 若只是普通店播口播，不应强行上升为“专业背书”。

---

## 4.4 CandidateSet

### 4.4.1 定义

CandidateSet 是商品诊断推理层在进入模块 4 之前，对外可下发的**候选表达集合**。

它回答的不是“这个商品本身是什么”，而是：
- 基于前序诊断，这个商品当前更适合走什么主说服路线；
- 可供下游脚本/结构生成模块优先调用的 H / E / C 候选池是什么；
- 哪些规则约束需要被同时带给下游模块。

它是**模块 3 的输出协议之一**，用于承接从商品事实向量到内容表达候选集的过渡。

### 4.4.2 字段清单

| 字段 | 类型 | 必填 | 含义 |
|---|---|---:|---|
| `product_task` | string | 是 | 商品核心任务，作为 CandidateSet 的一级锚点 |
| `main_persuasion_route` | string | 是 | 当前商品在该轮诊断中优先采用的主说服路线 |
| `r_rule` | string | 是 | R 侧规则结果，用于约束路线选择与表达边界 |
| `p_rule` | string | 是 | P 侧规则结果，用于约束人群/说服方式/表达落点 |
| `candidate_h_pool` | string[] | 是 | 候选 H 标签池，表示可优先尝试的 Hook 集合 |
| `core_e_list` | string[] | 是 | 候选核心 E 标签列表，表示主要 Effect 承接动作 |
| `core_c_list` | string[] | 是 | 候选核心 C 标签列表，表示主要 CTA 收口动作 |

### 4.4.3 字段定义与边界

#### `product_task`

定义：CandidateSet 所依附的商品核心任务。

作用：
- 保证候选表达集合与商品任务域保持一致；
- 防止 H / E / C 候选池脱离商品本体任务。

边界：
- 必须与 `product_fact_vector.product_task` 保持一致；
- 是 CandidateSet 的锚点字段，不是新增推理结论；
- 不得在 CandidateSet 中改写或扩写为脚本策略语句。

#### `main_persuasion_route`

定义：当前商品在本轮诊断下的主说服路线。

作用：
- 作为下游模块优先组织内容结构的主线；
- 为 H / E / C 候选选择提供统一路线约束。

边界：
- 必须是可执行的说服方向，不是抽象分析备注；
- 应体现“先讲什么、靠什么建立说服”的主路线，但不得直接写成长段脚本；
- 只能输出最终裁决后的主路线，不得把中间推理变量混入该字段。

#### `r_rule`

定义：R 侧规则的最终输出结果。

作用：
- 约束主说服路线的成立条件与边界；
- 为下游模块提供必须遵守的路线侧规则。

边界：
- 必须是规则结果，不是解释过程；
- 不得夹带策略推导链路全文；
- 不得与 `main_persuasion_route` 相互重复到仅剩改写关系。

#### `p_rule`

定义：P 侧规则的最终输出结果。

作用：
- 约束人群承接、表达方式、内容落点；
- 作为下游模块生成 H / E / C 时的并行约束条件。

边界：
- 必须是规则结果，不是开放式备注；
- 不得替代 `target_people` 或 `differentiator` 的商品事实字段；
- 不得写入模块外不可消费的中间变量。

#### `candidate_h_pool`

定义：当前商品可优先尝试的 Hook 候选集合。

作用：
- 为下游模块提供可选 H 起手库；
- 让后续生成不必从全量 H 标签盲选。

边界：
- 这是候选池，不是唯一最终标签；
- 元素应为 H 标签或其稳定协议值，不应混入口语说明句；
- 候选必须与 `main_persuasion_route`、`product_task` 保持一致。

#### `core_e_list`

定义：当前商品优先采用的核心 Effect 标签集合。

作用：
- 为下游模块提供主要的信息承接动作；
- 约束视频主体段的信息推进方向。

边界：
- 应是核心承接动作列表，而非所有可能 E 标签的穷举；
- 必须体现信息推进，不得只是同义重复；
- 不得把编排备注、镜头说明、导演意图写入该字段。

#### `core_c_list`

定义：当前商品优先采用的核心 CTA 标签集合。

作用：
- 为下游模块提供主要的收口动作候选；
- 约束结尾段如何完成行动承接。

边界：
- 应输出 CTA 标签或其稳定协议值，不得输出完整成交话术；
- 必须与渠道风险、商品类型、主说服路线相兼容；
- 若渠道不允许强逼单，应通过该字段显式收敛到安全 CTA 候选。

### 4.4.4 严格协议约束

以下字段或概念**只允许作为模块 3 的内部中间变量**，不得作为模块 4 的协议输出字段，也不得混入 CandidateSet：
- `category_strategy_intent`
- `product_strategy_intent`
- `intent_coordinates`
- `modifiers`
- 其他等价的“意图层中间推理变量”

原因：
- 这些字段属于诊断引擎内部推理态，不是下游稳定消费协议；
- 若透传到模块 4，会把中间意图层与最终候选集协议混淆；
- 会导致下游模块错误依赖内部变量，破坏模块边界稳定性。

因此，模块 4 允许消费的应当是**已经裁决完成的 CandidateSet 结果**，而不是上游意图推理过程本身。

### 4.4.5 与前三层的关系

- 4.1 / 4.2 / 4.3 解决的是“商品是什么、卖给谁、差异点是什么、事实属性是什么”；
- 4.4 解决的是“在这些事实约束下，下游内容模块优先使用什么候选表达集合”；
- 4.4 仍然不是最终脚本成稿，而是模块化生成前的候选协议层。

---

## 5. 三层之间的关系

### 5.1 从基础层到完整层

- 4.1 提供商品身份与原始卖点；
- 4.2 在 4.1 基础上，把商品主卖给谁、最核心差异点是什么明确出来。

### 5.2 从完整层到事实向量层

- 4.2 仍然偏自然语言表达；
- 4.3 则把商品抽象成可用于诊断引擎、规则系统、断言系统的结构化向量。

### 5.3 不能反向污染

- 4.3 不能反向生成 4.1 中不存在的商品身份信息；
- 4.2 不能写入策略话术；
- 4.1/4.2/4.3 任一层都不应直接出现视频级判断结果。

---

## 6. 与视频理解 / HEC 的边界

### 6.1 商品信息层回答什么

`product_info` 回答的是：
- 商品是什么；
- 卖给谁；
- 有什么主差异点；
- 它在任务、认知、频次、信任、价格、背书、渠道风险上的结构化属性是什么。

### 6.2 商品信息层不回答什么

`product_info` 不直接回答：
- 这条视频属于哪个 HEC 标签；
- 这条视频是否讲清楚了卖点；
- 该走什么脚本结构；
- 该用什么 CTA 动作。

这些属于后续视频理解与诊断输出层，不属于输入字典本身。

---

## 7. 强断言要求

以下情况必须 Crash Early：
- 缺少 `product_info`；
- 三层结构不完整；
- 用旧字段平铺替代三层结构；
- `leaf_category` 过粗；
- `price` 无法解析；
- `core_selling_points` 为空；
- `target_people` 或 `differentiator` 缺失；
- `product_fact_vector` 缺少任一核心属性；
- 把 HEC 标签、脚本策略、视频结论混入 `product_info`。

---

## 8. 推荐对外描述方式

### README / SKILL 简版表述

可对外简述为：

> `product_info` 必须由调用方显式传入，并按三层结构提供：基础商品信息、完整商品信息、商品事实向量。系统不会从视频或内部库静默补齐这些字段。

### Schema 文档表述

Schema 文档中建议只保留：
- 三层结构图；
- 字段名；
- 必填关系；
- 每层一句话定义；
- 指向本文档的链接。

## 10. 结论

商品诊断输入口径必须统一到四层结构：
- **4.1 基础商品信息**；
- **4.2 完整商品信息**；
- **4.3 商品事实向量**；
- **4.4 CandidateSet**。

后续无论是视频理解、商品诊断、脚本生成，还是 Schema、断言、QA，都必须以这四层结构作为唯一合法的 `product_info` / 商品诊断协议定义：前 3 层负责商品事实输入，4.4 CandidateSet 负责模块 3 输出给模块 4 的候选表达协议层。
